# Shiroha Quiz 后续功能开发计划 32-33

当前版本：原生 `v0.6.6-native`，Web `v0.4.6-alpha`。Web 壳进入稳定维护，原生 Compose 版作为主要开发线继续推进。

---

## 一、当前阶段定位

32 阶段结束后，原生 Compose 版已具备完整的学习闭环。阶段 32-33 的核心任务是：在稳定现有功能的基础上，深挖导入体验、增强刷题能力、补齐工程基础设施。

当前核心链路已具备：

- 题库导入（docx/txt/xlsx/json）、多策略解析、核对、写入；
- 普通练习、批量练习、即时判题、自动下一题；
- 考试、计时、交卷、记录明细；
- 错题本、连续掌握规则、错题复习；
- 背题模式、斩题功能、题目收藏、快速编辑；
- 智能复习模式、首页待复习提示；
- 字号缩放、紧凑选项、顺序练习进度记忆；
- 首页紧凑布局、显示异常设备兼容；
- 双图标系统、Shiroha 模式（开屏/插画/图标开关）；
- 素材 WebP 化、暗夜/浅色模式、Design Token 系统；
- AI 核对与 AI 解析（OpenAI 兼容）；
- Excel 表格导入、数学/化学符号识别；
- JSON 跨端互导、图片题互通（ZIP + base64）；
- 导入编辑器查找替换（正则）；
- R8 + 资源收缩 + 正式签名。

因此 32-33 阶段不建议大范围重构，而应优先围绕三个方向推进：

1. **导入体验深挖**：强化现有导入链路的容错与效率。
2. **刷题体验增强**：补齐填空题、简答题的作答与判分能力。
3. **工程与性能**：为大题库场景和持续集成打好基础。

## 二、32 阶段前已完成能力清单

以下能力在 32 阶段前已经落地，后续阶段不得回退：

### 1. 视觉与主题

- 双图标系统统一至 native 源集；
- Shiroha 模式（开屏图 / 插画 / 图标统一开关）；
- 素材全面 WebP + 暗夜模式无边框；
- 暗夜 / 浅色模式切换 + Design Token 系统。

### 2. 导入链路

- 题库 JSON 相互导入，含图片完全互通（ZIP + base64）；
- 导入编辑器查找替换（正则）；
- 填空题识别优化（关键词扩展，减少简答题误判）；
- 题干中包含"格式"时的分区误识别修复；
- 核对页筛选器按需显示 + 自动聚焦；
- 单题编辑页 + AI 建议前置；
- Excel 表格导入 + 数学 / 化学符号识别。

### 3. 练习与考试

- 练习 / 考试主流程稳定（答题、着色、收卡、滑动切题、答题卡）；
- 背题模式、斩题功能、题目收藏和题目快速编辑；
- 智能复习模式、错题掌握状态和首页待复习数量提示；
- 字号缩放、紧凑选项、顺序练习进度记忆，退出练习时可保存当前位置。

### 4. 兼容性

- 首页紧凑布局和显示异常设备兼容，题目文本允许自动换行；
- 二级页面系统返回映射，避免退回桌面。

### 5. 工程

- AI 核对与解析（OpenAI 兼容）；
- R8 + 资源收缩 + 正式签名。

## 三、32-33 阶段开发目标

### 目标 1：导入体验深挖

优先处理：

- 深化 Excel 表格导入的列映射灵活性（混合布局、合并单元格）；
- 继续核校真实题库导入结果，沉淀更多解析样例；
- 优化大题库（500+ 题）导入预览性能。

可选方向：

- 导入预览异常自动批量修复建议。

验收标准：

- 标准 docx/txt/xlsx/json 解析不被破坏；
- 大题库导入不出现 ANR 或明显卡顿；
- 解析样例库持续积累。

---

### 目标 2：刷题体验增强

优先处理：

- 填空题答案归一化（等价答案 / 多表述匹配）；
- 简答题作答输入与参考答案展示。

可选方向：

- 智能复习策略继续细化，例如复习间隔、到期提醒和掌握状态解释。

验收标准：

- 填空题支持多答案分隔和标点归一化自动判分；
- 简答题可输入作答，提交后展示参考答案，不自动判分；
- 不影响现有选择题、判断题的正确率统计。

---

### 目标 3：工程与性能

优先处理：

- Room 持久化替代 SharedPreferences JSON（大题库场景）；
- 批量导入性能方案。

可选方向：

- 单元测试补齐（解析器为核心）；
- CI 自动编译 + APK 产出。

验收标准：

- 大题库读写性能不劣于 SharedPreferences 方案；
- 解析器核心路径有测试覆盖。

---

## 四、Web 壳维护策略

Web 壳作为稳定对照版本，仅做必要维护（安全修复、重大兼容问题）。功能开发以原生版为主，Web 版只同步必要的数据兼容和严重问题修复。

后续注意：

- 原生端新增字段时必须考虑 JSON 导出与 Web 端互认；
- Web 端功能迭代不影响原生端的核心开发节奏；
- 两端题库格式保持兼容，不做破坏性变更。

---

## 五、建议下一轮开发顺序

建议顺序：

1. **填空题答案归一化** — 支持多答案分隔、标点归一化、自动判分；
2. **简答题作答** — 输入框作答、参考答案展示、不自动判分；
3. **导入体验优化** — Excel 列映射增强、大题库性能；
4. **工程基础设施** — Room 迁移、单元测试、CI。

---

## 六、开发约束

后续开发必须遵守：

1. **只负责原生 Android / Compose** — 默认开发范围 `apps/android/app/src/native/`；
2. **最小修改原则** — 不顺手重构，不引入无关变更；
3. **不回退既有功能** — 已完成能力清单中的所有功能不得被破坏；
4. **标准解析优先** — 标准题库解析主线必须稳定，复杂兜底不能污染标准格式；
5. **敏感信息隔离** — 代码、测试样例、文档中不得写入真实组织、人名等敏感信息。
