## 一、当前阶段定位

31.5 之后，原生 Compose 版已经从“核心流程搭建”进入“真实使用体验打磨”阶段。

当前核心链路已经具备：

- 题库导入、解析、核对、写入；
- 普通练习、批量练习、即时反馈；
- 考试、计时、交卷、记录；
- 错题本、错题复习、掌握状态；
- 背题模式；
- 斩题功能；
- JSON 题型大小写兼容；
- 图片题 JSON 导入兼容；
- 长选项完整显示；
- 二级页面系统返回映射；
- 暗夜模式、Shiroha 模式、基础视觉 Token；
- AI 核对与 AI 解析基础能力。

因此 31.5-32 阶段不建议再大范围重构，而应优先围绕三个方向推进：

1. **稳定性收口**：避免新功能污染标准导入、标准练习和错题本主线。
2. **学习闭环增强**：让背题、斩题、错题、记录、掌握状态之间的关系更清楚。
3. **发布前体验完善**：补齐页面说明、空状态、异常提示、导入失败兜底和实机细节。

## 二、当前已完成能力交接

### 1. 二级页面系统返回映射

已完成：

- 非底部一级 Tab 页面由原生 `BackHandler` 接管系统侧边返回；
- 避免从二级页面直接退回桌面；
- 一级底部 Tab 页面仍保持系统默认返回行为。

当前设计：

| 当前页面                                   | 系统返回目标 |
| -------------------------------------- | ------ |
| AI 设置 / 外观偏好 / 刷题偏好 / 数据管理 / 标准格式 / 关于 | 我的     |
| 记录详情                                   | 记录     |
| 题库核对                                   | 题库详情   |
| 题库管理 / 题库详情 / 错题本 / 记录 / 考试            | 首页     |
| 首页 / 练习 / 导入 / 我的                      | 系统默认返回 |

后续注意：

- 新增页面时必须同步补充系统返回映射；
- 不要只处理顶部返回按钮，必须同时考虑 Android 系统侧边返回；
- 页面跳转层级不要过深，避免出现无法预期的返回链。

### 2. 背题模式

已完成：

- 入口：`我的 → 刷题偏好 → 背题模式`；
- 作用范围：仅普通练习页；
- 开启后练习页直接显示答案和解析；
- 不需要提交答案；
- 不计入正确率；
- 不加入错题；
- 不生成普通练习记录；
- 不影响考试模式。

后续注意：

- 背题模式是“看题学习”，不是正式答题；
- 所有统计、错题、记录、正确率逻辑都不应被背题模式污染；
- 后续如新增背题记录，也应单独建模，不要复用正式练习记录。

---

### 3. 斩题功能

已完成：

- 入口：`我的 → 刷题偏好 → 斩题功能`；
- 练习页题目卡片右上角显示小圆形“斩”按钮；
- 点击后将题目加入已斩题集合；
- 当前普通练习中移除该题；
- 后续普通练习默认跳过已斩题；
- 题库详情页显示“斩 X”入口；
- 题库详情内可恢复单题。

设计边界：

- 斩题不是删除题目；
- 斩题只影响普通练习池；
- 斩题不影响考试；
- 斩题不等于错题本已掌握；
- 错题本已掌握也不自动等于斩题。

后续注意：

- 后续如增加“批量恢复斩题”，应放在题库详情或数据管理中；
- 不建议在考试页显示斩题入口；
- 不建议让斩题动作修改错题统计。

---

### 4. 即时判题与自动下一题拆分

已完成：

- 刷题偏好拆成两个开关：
  - 选择后立即判题；
  - 判题后自动下一题。

当前行为：

| 开关状态         | 行为                    |
| ------------ | --------------------- |
| 立即判题开，自动下一题关 | 点选项后显示对错、答案、解析，但停留在本题 |
| 立即判题开，自动下一题开 | 点选项后判题，短暂停留后进入下一题     |
| 立即判题关        | 手动提交                  |

作用范围：

- 只作用于即时练习；
- 不作用于批量刷题；
- 不作用于背题模式；
- 不作用于考试模式。

后续注意：

- 批量刷题仍保持“选完一组，手动提交本组，统一看结果”；
- 自动下一题的停留时间建议保持短而稳定，不要加入复杂动画；
- 如果未来加入“答错自动停留更久”，应作为单独偏好，不要改变当前默认行为。

### 5. 错题本掌握规则

已完成：

- 错题本显示仍为历史累计：`错 X 次 / 对 Y 次`；
- 已掌握判定改为连续正确：连续答对 2 次 → 已掌握；
- 答错一次：`wrongCount + 1`，连续正确次数清零，状态回到未掌握；
- 手动标记掌握不再强行修改历史正确次数；
- 手动取消掌握不再清空历史正确次数。

当前口径：

| 项目                   | 含义                     |
| -------------------- | ---------------------- |
| `wrongCount`         | 进入错题本后的累计答错次数          |
| `rightCount`         | 进入错题本后的累计答对次数          |
| `reviewRightCount`   | 复习路径中的正确次数，作为历史展示或辅助统计 |
| `streakCorrectCount` | 连续答对次数，用于掌握判定          |
| `status`             | 当前是否已掌握                |

计数范围：

- 计入：普通练习、刷错题、批量练习提交、考试提交；
- 不计入：背题模式、斩题动作、纯浏览动作。

后续注意：

- 历史累计展示和掌握判定必须分离；
- 不要再恢复“累计答对 3 次掌握”的旧规则；
- 后续如增加“掌握题复习”，应明确是否展示已掌握题，而不是改变错题本基础判定。

影响范围：

- 练习页；

### 7. JSON 题型与图片兼容

已完成：

题型兼容：

- `single / SINGLE` → 单选；
- `multiple / MULTIPLE` → 多选；
- `judge / JUDGE` → 判断；
- `blank / BLANK` → 填空；
- `short / SHORT` → 简答。

别名兼容：

- `choice / radio` → 单选；
- `multi / checkbox` → 多选；
- `true_false / boolean` → 判断；
- `fill_blank` → 填空；
- `essay / qa / subjective` → 简答。

图片兼容：

- 支持旧 Web 导出的 Markdown base64 图片；
- 支持新 Web 导出的 `images` 数组；
- 支持 `dataUrl / dataUri / localPath` 等字段；
- 导入后进入原生题图展示链路。

后续注意：

- 兼容逻辑应放在导入边界，不要污染内部题库结构；
- 新增字段时尽量做宽容读取；
- 图片导入失败时应给出用户可理解的提示，而不是静默丢图。

## 三、31.5-32 阶段开发目标

### 目标 1：稳定性收口

优先处理：

- 本地 `compileNativeReleaseKotlin` 编译验证；
- 实机系统返回验证；
- 背题、斩题、考试三者边界验证；
- 错题本连续掌握规则验证；
- 长选项滚动验证；
- 图片题导入与展示验证。

验收标准：

- 不出现 Kotlin 编译错误；
- 不出现二级页面侧边返回直接退桌面；
- 背题模式不产生错题、不产生正式练习记录；
- 斩题不影响考试；
- 错题本连续答对 2 次掌握，答错后回未掌握；
- 长选项完整显示，不截断、不遮挡；
- 图片题导入后不显示 base64 文本。

---

### 目标 2：刷题偏好说明完善

建议增加或优化：

- 背题模式说明文案；
- 斩题功能说明文案；
- 选择后立即判题说明；
- 判题后自动下一题说明；
- 错题掌握规则说明。

建议文案方向：

- 背题模式：适合先看答案和解析，不计入正式练习统计；
- 斩题功能：把已经很熟的题临时移出普通练习池，不会删除题目；
- 立即判题：选择答案后立刻显示正误；
- 自动下一题：判题后短暂停留并自动进入下一题；
- 错题掌握：连续答对 2 次才会自动掌握，答错会重新进入未掌握状态。

验收标准：

- 用户能在不看 README 的情况下理解每个开关的作用；
- 不出现“斩题 = 删除题目”的误解；
- 不出现“背题也会计入正确率”的误解；
- 不出现“已掌握会自动斩题”的误解。

---

### 目标 3：错题本体验增强

可选方向：

1. 错题本卡片补充连续正确状态：
   
   - 显示 `连续对 0/2`、`连续对 1/2`、`已掌握`；
   - 帮助用户理解为什么答对一次还没掌握。

2. 错题筛选增强：
   
   - 未掌握；
   - 已掌握；
   - 全部错题。

3. 已掌握题的处理说明：
   
   - 明确已掌握题不会从题库删除；
   - 明确已掌握题是否出现在刷错题入口。

验收标准：

- 错题状态可解释；
- 统计数字不被状态操作篡改；
- 刷错题入口行为与页面说明一致。

---

### 目标 4：斩题管理完善

可选方向：

1. 题库详情内增加斩题管理空状态：
   
   - 没有斩题时显示说明；
   - 有斩题时支持单题恢复。

2. 增加批量恢复：
   
   - 恢复当前题库所有斩题；
   - 操作前二次确认。

3. 练习页斩题后反馈优化：
   
   - Toast 或轻提示说明“已移出普通练习池”；
   - 当前练习池为空时给出明确提示。

验收标准：

- 用户知道斩题不会删除题目；
- 用户能恢复误斩题；
- 当前练习池被斩空时不会出现空白页或异常跳转。

---

### 目标 5：导入与图片题稳定性增强

当前原则：

- 标准题库解析主线优先；
- 复杂兜底只能低优先级启用；
- 不能为了复杂真题污染标准解析结果。

建议方向：

1. 图片导入失败提示：
   
   - base64 解析失败；
   - 图片落地失败；
   - 图片字段存在但内容为空。

2. 图片题展示兜底：
   
   - 图片缺失时显示轻提示；
   - 不把图片路径或 base64 文本暴露给用户。

3. JSON 导入结果预检：
   
   - 显示题型统计；
   - 显示图片题数量；
   - 显示异常题数量。

验收标准：

- 标准 docx/txt/xlsx/json 解析不被破坏；
- Web 导出的图片题原生导入后正常显示；
- 图片失败时用户能知道问题位置；
- 不出现题干里直接展示超长 base64 的情况。

---

### 目标 6：记录页与学习轨迹增强

可选方向：

1. 记录详情补充模式来源：
   
   - 普通练习；
   - 批量练习；
   - 考试；
   - 错题复习。

2. 今日学习状态更准确：
   
   - 今日练习题数；
   - 今日正确率；
   - 今日错题新增；
   - 今日复习错题。

3. 背题模式是否单独记录：
   
   - 当前不建议进入正式练习记录；
   - 如果要记录，应单独作为“学习浏览记录”，不混入正确率。

验收标准：

- 记录页不混淆背题和正式答题；
- 错题复习能回溯；
- 首页学习状态和记录页数字逻辑一致。

### 目标 7：AI 功能收口

当前已有：

- AI 设置；
- OpenAI 兼容接口；
- API Key / 模型 / 地址配置；
- AI 核对；
- AI 解析；
- 异常题优先处理；
- 分批处理。

建议方向：

1. AI 设置页说明增强：
   
   - 服务商选择说明；
   - 自定义接口填写说明；
   - 超时、批量题数、保存阈值说明。

3. AI 失败提示：
   
   - 网络失败；
   - 鉴权失败；
   - 响应格式异常；
   - 超时。

验收标准：

- AI 配置错误时提示清楚；
- AI 建议不会绕过人工核对直接改正式库；
- 标准导入流程不依赖 AI。

### 2. 背题模式验收

2. 进入练习页；
3. 查看是否直接显示答案和解析；
4. 不提交答案直接切题；
5. 检查是否生成练习记录；

预期：

- 不新增错题；

---

### 3. 斩题功能验收

步骤：

1. 开启斩题功能；
2. 进入普通练习；
3. 点击题目右上角“斩”；
4. 重新进入普通练习；
5. 进入考试；
6. 进入题库详情斩题管理并恢复。

预期：

- 被斩题不再进入普通练习池；
- 考试仍可能出现该题；
- 题库详情可恢复；
- 不影响错题统计。

---

### 4. 即时判题验收

步骤：

1. 开启立即判题，关闭自动下一题；
2. 选择答案；
3. 开启自动下一题；
4. 再次选择答案；
5. 关闭立即判题；
6. 测试手动提交。

预期：

- 立即判题开启时，点选项后显示正误；
- 自动下一题关闭时停留本题；
- 自动下一题开启时短暂停留后切题；
- 立即判题关闭时必须手动提交。

---

### 5. 错题掌握验收

建议构造一题：

1. 先答错 → 进入错题本；
2. 答对 1 次 → 仍未掌握；
3. 再答对 1 次 → 已掌握；
4. 再答错 1 次 → 回到未掌握，连续正确清零；
5. 再连续答对 2 次 → 再次已掌握；
6. 手动标记掌握 / 取消掌握 → 检查历史 `对 Y 次` 不被强行改成 3 或 0。

预期：

- 掌握规则只看连续正确次数；
- 历史累计展示不被手动状态操作篡改。

---

### 6. 长选项验收

步骤：

1. 导入或构造长选项题；
2. 进入练习页；
3. 进入考试页；
4. 检查选项卡片。

预期：

- 长选项完整换行；
- 不出现省略号；
- 页面可正常滚动；
- 短选项视觉不松散。

---

### 7. 图片 JSON 验收

步骤：

1. 使用含图片题的 Web 导出 JSON；
2. 导入原生端；
3. 检查题型统计；

## 六、开发约束

后续开发必须遵守：

1. **只负责原生 Android / Compose**
   
   - 默认开发范围：`apps/android/app/src/native/`；
   - 不处理 `apps/web/`；
   - 不处理 `apps/android/app/src/main/assets/web/`。

2. **最小修改原则**
   
   - 只改任务相关文件；
   - 不顺手重构；
   - 不引入无关 UI 或业务变更。

3. **不回退既有功能**
   
   - 背题、斩题、即时判题、错题规则、图片兼容等必须继续保留；
   - 新功能必须基于最新源码继续叠加。

4. **标准解析优先**
   
   - 标准题库解析主线必须稳定；
   - 复杂真题兜底逻辑只能低优先级启用；
   - 不允许为了复杂样例污染标准格式。

5. **敏感信息隔离**
   
   - 代码、测试样例、README、文档中不得写入真实内部组织、项目部、中队、文件编号、作业编号、人名等敏感信息；
   - 示例题库必须匿名化、通用化、虚构化。

6. **交付格式**
   
   - 默认交付最小替换包；
   - 压缩包目录从 `apps/` 开始；
   - 只包含实际修改文件；
   - 提供覆盖说明、改动要点、中文提交信息。
