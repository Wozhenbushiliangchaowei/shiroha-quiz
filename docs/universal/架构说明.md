# 架构说明

## 1. 总体思路

本项目采用单仓多项目结构，也就是常见的 `monorepo` 思路。

原因很简单：

- Web 和 Android 未来都会围绕“题库导入、刷题、考试、错题、记录”这些核心逻辑展开
- 如果一开始就拆成多个仓库，后期同步成本会更高
- 如果把所有内容混在一个目录里，后面边界会越来越乱

所以这里采用：

- 一个仓库
- 多个清晰子目录
- 后续把共享逻辑慢慢沉淀到 `packages`

## 2. 目录职责

### apps/web

当前通用网页版参考实现。

主要职责：

- 题库导入能力验证
- 导入策略验证
- 题库格式参考
- Web 侧功能演示

### apps/android

安卓原生应用主工程。

主要职责：

- 原生 UI 和交互
- 刷题体验
- 考试体验
- 本地记录与状态管理

### packages/core

后续沉淀核心刷题逻辑。

例如：

- 答案判断
- 计分
- 错题记录
- 刷题会话管理

### packages/parser

后续沉淀题库解析逻辑。

例如：

- 文本导入
- 题型识别
- 选项标准化
- 双文件合并

### packages/types

后续沉淀通用数据结构。

例如：

- Question
- QuestionBank
- AnswerRecord

### packages/shared

后续沉淀通用工具函数。

### packages/ui

后续沉淀可复用的 UI 规范或共享设计基础。

### docs

放产品、设计、页面、架构、执行计划等中文文档。

### assets

放图标、宣传图、设计素材。

## 3. 当前实现策略

采用”边开发边抽离”的方式，避免过早抽象。两端独立演进，共享核心解析逻辑。

### 安卓端（当前主开发线）

已全面迁移至 Kotlin + Jetpack Compose + Material3，产品风味构建：

| Flavor | 包名 | 说明 |
|--------|------|------|
| `native` | `com.reqir.shirohaquiz` | 原生 Compose 版，当前主线 |
| `web` | `com.yiqiu.shirohaquiz` | WebView 壳版，稳定维护 |

原生版已落地 14 个页面、17 个解析器组件，覆盖练习/考试/错题/收藏/记录/AI 分析/导入全流程。

### Web 端

单页应用，纯前端实现。解析器侧重原文区间修复和脏文本容错，与原生端共享核心解析规则。

## 6. 后续方向

- 原生导入解析持续核校更多真实题库

- 后台化模块（题库分发、用户管理）持续开发
- 大题库场景性能优化与测试体系建设
